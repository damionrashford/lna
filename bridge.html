The stdio bridge

LNA lets a public page open a socket to localhost. This page uses that socket to drive a real process's stdin/stdout/stderr on your machine — the pattern that bridges a browser agent to a local shell or a stdio MCP server. LNA gates the hop; the local daemon does the stdio translation.

flow: this page (public https) ──WebSocket──▶ bridge daemon (127.0.0.1:7967) ──stdin/stdout──▶ spawned process LNA gates ▲ this hop only the daemon spawns + pipes ▲ bash · python3 · an MCP server · any allowlisted binary

Run the bridge daemon first

  1. curl -sO https://raw.githubusercontent.com/damionrashford/lna/main/bridge-server.ts
  2. BRIDGE_TOKEN=dev bun bridge-server.ts → listens on 127.0.0.1:7967, prints its token + command allowlist.
  3. For real LNA enforcement, load this page from the public site (not localhost) with chrome://flags/#local-network-access-check set to Enabled (Blocking), then connect below and grant the prompt.
Why the token + allowlist: a spawn endpoint reachable from a public origin is remote code execution. The daemon only launches allowlisted binaries and only after a token handshake. Keep the token secret if you ever front it with a tunnel.
