1.0

Threat model

why the browser is stepping in

RFC 1918 has separated private from public addresses for three decades, but browsers never enforced the boundary: any public website can make your browser fire requests at devices inside your network. The Local Network Access spec cites the documented consequences — drive-by pharming, SOHO router pharming, and an exploit kit dedicated to CSRF that rewrote router DNS settings at scale.

Why CORS doesn't stop this

The attack request is a CORS-safelisted method with safelisted headers, so no preflight fires and the target device is never asked. The attacker doesn't need to read the response — the request itself is the CSRF attack. Two more realities widen the blast radius:

  • Localhost web servers are now a common way to ship desktop software, and they routinely assume protections that don't exist — see Project Zero's Avast "Avastium" and Trend Micro cases, where a web-reachable local endpoint meant remote code execution.
  • Fingerprinting and tracking — sites probe your local network to identify you, including the "Local Mess" technique where a localhost endpoint and a remote site collude to link identities.

LNA's answer: deprecate silent access and gate every public → local request behind a user permission. Chrome first tried the inverse — Private Network Access, where the device had to opt in via CORS preflights — and put it on hold after rollout proved impractical. LNA replaces it, asking the user instead.

2.0

Address spaces

the model everything hangs on

Every IP address belongs to exactly one of three address spaces, defined by spec §2.1:

public

Same target for everyone

Reachable from anywhere on the internet — e.g. 104.18.27.120. Everything not in the table below.

local

Same target per network

Meaningful only inside the current network — e.g. 192.168.0.1. Your router, printer, NAS.

loopback

Different target per device

Only reachable on the local host — e.g. 127.0.0.1, aka localhost. Software on the user's own machine.

The spec defines a "less public" ordering: loopback < local < public. A request that crosses down this ordering is what triggers LNA.

How an address is classified

Per the spec's algorithm: if the address is an IPv4-mapped IPv6 address (::ffff:0:0/96), unwrap the embedded IPv4 address first; then look it up in the non-public blocks table; anything unmatched is public.

3.0

Permission model

what triggers, what gates, what delegates

A local network request (spec §2.2) is any request whose target resolves to an address space less public than the requesting context's own space. In Chrome's first milestone that means: public → local and public → loopback. Requests originating from loopback are exempt — software on the machine is already the most privileged vantage point.

When such a request is detected, the browser prompts the user. Grant → the fetch proceeds. Deny → it fails with a network error. Permission scope is implementation-defined (per-origin at minimum) and the decision may persist to reduce prompt fatigue.

Two permissions, one legacy alias

The spec defines two default powerful features, each a policy-controlled feature with default allowlist 'self':

  • local-network — gates requests to local addresses
  • loopback-network — gates requests to loopback addresses
  • local-network-access — the original coarse name, kept as a backwards-compatible alias covering both

Querying the alias combines the two underlying states — any denied wins, then any allowed, else prompt (full 9-row table on MDN).

Secure contexts only

Making local network requests is a powerful feature restricted to secure contexts. On plain-HTTP pages, gated requests simply fail — one more reason the initiating page must be HTTPS.

Delegating into iframes

Cross-origin iframes get no local network access by default. Delegate with the Permissions-Policy header or the allow attribute:

Permissions-Policy: local-network=(self "https://example.com")

<iframe src="https://example.com/setup" allow="local-network"></iframe>

The embedded document's request is treated as if the embedding document asked — the user's decision binds to the top-level origin. If the iframe navigates across origins that also need access, list every origin, or use allow="local-network *" for redirect chains (SSO flows that bounce before returning to localhost).

The CSP escape hatch

A document served from the local network can voluntarily drop its privilege with the non-standard CSP directive treat-as-public-address — it is then treated as public for LNA purposes. Ignored in report-only mode and in <meta>.

4.0

API surface

what you actually write

The fetch extension

LNA adds one enum and one option to Fetch (spec §3.1.2):

enum IPAddressSpace { "public", "local", "loopback" };

partial dictionary RequestInit {
  IPAddressSpace targetAddressSpace;
};

partial interface Request {
  readonly attribute IPAddressSpace targetAddressSpace;  // "unknown" if unset
};

Mixed content, relaxed — deliberately

Local devices largely can't get publicly-trusted TLS certificates, so requiring HTTPS pages and blocking HTTP subrequests would strand them. LNA therefore exempts a request from mixed-content blocking when the browser can tell it's local before DNS resolution:

// 1 — private IP literal: exempt
fetch("http://192.168.0.1/ping");

// 2 — .local domain: exempt
fetch("http://router.local/ping");

// 3 — public hostname: NOT exempt, even if it resolves locally
fetch("http://example.com/ping");

// 4 — explicit annotation: exempt
fetch("http://example.com/ping", { targetAddressSpace: "local" });

Case 4 is the one to remember: hostnames like http://internal.example.com that resolve to private addresses need the annotation. It can't be abused as a general mixed-content bypass — if the connection's resolved IP isn't in the declared address space, the request fails.

Checking before you fetch

const status = await navigator.permissions.query({ name: "local-network" });
// status.state → "granted" | "prompt" | "denied"

const req = new Request("http://localhost:8888", {
  method: "get",
  mode: "cors",
  targetAddressSpace: "loopback",
});
fetch(req);

Everything else that's gated

  • Subresources and iframe navigations — gated today. Main-frame navigations are not (yet); the spec notes popups may deserve it.
  • WebSockets — gated via the same fetch handshake; note there is no targetAddressSpace equivalent for ws:// URLs, so no mixed-content bypass exists there.
  • WebTransport — gated where it obtains connections through Fetch.
  • Workers — a worker's address space comes from its script response's policy container. Service-worker fetches are checked against the worker's script origin; the SW soft-update algorithm's null client is a known open issue.
  • HTTP cache — cached responses remember the IP they were loaded from, and LNA re-checks on every cache hit. A denied request caches nothing.

Under the hood: the check runs after a connection is obtained, per connection, because DNS answers change (see 5.0). With Happy Eyeballs, hosts with both public and private A/AAAA records can pass or fail non-deterministically.

5.0

Limits & caveats

what LNA does not solve

The spec is unusually candid about its own edges (§5, Security & Privacy):

  • DNS rebinding — the check runs on the IP actually connected to, on every new connection, or an attacker could re-point a hostname at a private address after approval.
  • Cross-network confusion — the permission isn't scoped to a network. Grant on the café Wi-Fi, and the grant follows you home: a captive-portal page left open (or restored from cache) can fire CSRF at your home devices. The spec calls network-scoped grants a possible future mitigation.
  • Local-network attackers — Chrome's first milestone gates only public→local. A malicious device already on your LAN (or a captive portal) can still probe localhost; the spec leaves local→local and local→loopback enforcement as a future increment.
  • Not absolution for device vendors — a router admin panel must still ship its own CSRF defenses. LNA mitigates; it doesn't solve.
  • Cache probing & poisoning — cached-subresource checks stop a public site from sniffing which private sites you've visited; DNS rebinding + plaintext HTTP can still poison caches. Partitioning limits, not eliminates, both.
  • Proxies — proxied resources take the proxy's address space; a proxied document can discover it's being proxied by noticing local access works.
  • File URLs — treated as local for now, pending implementation experience.
  • Classification is a heuristic — an intranet on public IPs (false positive) or a VPN + home-network split (false negative) both break the two-space assumption. The spec accepts this as a pragmatic trade.

7.0

Prepare your site

the migration checklist
  1. Reproduce the future, today

    Set chrome://flags/#local-network-access-check to "Enabled (Blocking)" and walk every flow that touches a device, an intranet host, or localhost. Verify the prompt fires and the flow survives a grant.

  2. Serve the initiating page over HTTPS

    The permission only exists in secure contexts. HTTP pages lose local access entirely once LNA ships (the planned Origin Trial only buys time).

  3. Annotate fetches to public hostnames that resolve privately

    Add targetAddressSpace: "local" (or "loopback") to those calls, or they'll die as mixed content. IP literals and .local names don't need it.

  4. Delegate into iframes explicitly

    allow="local-network" on the frame, plus Permissions-Policy on your responses if you restrict origins. List every origin the frame navigates through.

  5. Query before you prompt

    Use navigator.permissions.query({ name: "local-network" }) to detect a prior deny and show your own remediation UI instead of failing silently.

  6. Mind the workers

    Service-worker fetches need a prior grant to the worker's origin — trigger one local request from a document context first, until worker-initiated prompting lands.

  7. Managed fleets: wait for the policy

    Enterprise deployments will be able to pre-grant or pre-deny per site — no prompts for known-good internal tools.

  8. Report what breaks

    File at the Chromium tracker or the spec repo — the Chrome team is explicitly asking for real-world use cases.
